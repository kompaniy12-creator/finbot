# Incident Response Runbook (FinBot)

Что делать при подозрении на утечку данных или компрометацию. Цель: быстро
ограничить ущерб, оценить масштаб, восстановиться, уведомить (GDPR - 72 часа).

## Контакты и роли

- Владелец/админ: Серхий (Telegram admin id в `TELEGRAM_ADMIN_TELEGRAM_ID`).
- Инфраструктура: Supabase (проект `bltbuptzsswaislqagwe`, регион eu-west-3),
  GitHub (репозиторий + Actions), Telegram BotFather.
- Эскалация: при утечке финансовых данных - разовая консультация юриста (DPA,
  уведомление субъектов и регулятора).

## Сигналы инцидента

- Алерт «всплеск неудачных аутентификаций вебхука» (приходит админу в бота).
- Аномалии в `security_audit` (много `export`/`key_*`/доступов за короткое время).
- Жалоба пользователя на чужие данные.
- Срабатывание gitleaks в CI (секрет в коммите).

## Шаги реагирования

### 1. Локализация (минуты)

- Скомпрометирован токен бота → BotFather `/revoke`, выпустить новый, обновить
  Supabase secret (`TELEGRAM_BOT_TOKEN[_SAAS]`), `setWebhook` заново.
- Скомпрометирован `SUPABASE_SERVICE_ROLE_KEY`/secret key → ротация ключей в
  Supabase (Settings → API keys), обновить Edge secrets, передеплой.
- Скомпрометирован KEK или `KEY_ENC_SECRET` → см. раздел «Ротация ключей».
- Подозрение на чужой доступ → отозвать доступ тестера через `/invites`
  (suspend tenant + деактивация участников); при необходимости отключить бота
  (снять webhook).

### 2. Оценка масштаба (по аудиту)

- Прочитать `security_audit` за период: какие действия, какие тенанты, результат.
- Прочитать `anthropic_usage` / `expense_audit` на предмет аномального доступа.
- Определить, какие тенанты и какие данные затронуты.

### 3. Ротация ключей

- **Токены ботов / API-ключи Supabase**: ротация в соответствующей панели +
  обновление Edge secrets + передеплой.
- **KEK (мастер-ключ шифрования)**: сгенерировать новый, положить в Vault,
  переобернуть все DEK новым KEK (скрипт переобёртки - TODO), либо, как временная
  мера, перешифровать данные `scripts/rotate_keys.ts` после смены DEK.
- **DEK конкретного тенанта**: пометить старый неактивным, перешифровать данные
  этого тенанта (`scripts/rotate_keys.ts --tenant <id>`).
- **Крипто-шреддинг**: если ключи тенанта надо немедленно сделать нечитаемыми -
  удалить его строки в `tenant_deks` (см. `shredTenantKeys` / `/delete_keys`).

### 4. Восстановление

- Данные: восстановление из зашифрованного бэкапа (age) - см. «Бэкапы».
- Если БД цела, а KEK утерян: `v2`-поля (API-ключи) нечитаемы - пользователи
  вводят ключи заново. Финансовые данные (не зашифрованы пер-тенантным DEK)
  доступны. Поэтому **офлайн-бэкап KEK обязателен**.

### 5. Уведомление (GDPR)

- При утечке персональных/финансовых данных - уведомить субъектов и, при
  необходимости, регулятора в течение 72 часов (юридическая часть - консультация).
- Подготовить краткое описание: что, когда, сколько субъектов, какие меры приняты.

## Бэкапы (P2.1)

- `cron-backup` дампит таблицы FinBot, шифрует `age` (`BACKUP_ENCRYPTION_KEY`),
  кладёт в приватные GitHub Releases. Гейт: `system_health.backup_key_confirmed`.
- Восстановление: `AGE_PRIVATE_KEY=... deno run scripts/restore.ts < backup.age`
  (приватный age-ключ - в твоём 1Password, один раз показан при сетапе).
- **Тест восстановления** проводить периодически на отдельной БД.

## Профилактика

- 2FA включить на: Supabase, GitHub, BotFather/Telegram, менеджер секретов.
- Принцип наименьших привилегий для сервисных учёток.
- Не коммитить секреты (pre-commit + gitleaks в CI ловят).
