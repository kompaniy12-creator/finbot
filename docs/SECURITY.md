# Безопасность данных FinBot

Этот документ честно описывает, как защищены данные пользователей: что зашифровано,
как разделяются пользователи, и где реальные границы (без маркетинга).

## Где хранятся данные

- **База:** один проект Supabase (PostgreSQL, регион ЕС). Разделение между
  пользователями - логическое, по `tenant_id` (см. «Изоляция»).
- **Фото чеков:** Supabase Storage, бакет `receipts` - приватный (`public=false`),
  доступ только по подписанным ссылкам с TTL 5 минут; авто-удаление через 90 дней.
- **Секреты сервера** (токены ботов, service-role): в Supabase Secrets, не в git
  (пре-коммит сканер секретов).
- **Бэкапы:** шифруются `age` (асимметрично) в приватные GitHub Releases.

## Изоляция пользователей (тенантов)

Каждый пользователь видит только свои данные. Три слоя защиты:

1. **`tenant_id` + обёртка `tenantDb()`** - все запросы автоматически фильтруются по
   тенанту, вставки им штампуются. Безопасный путь - по умолчанию.
2. **CI-страж** (`tenant_scoping_guard`) - сборка падает, если кто-то обратится к
   пер-тенантной таблице в обход обёртки.
3. **RLS (Row Level Security)** включён и форсирован на всех таблицах с данными
   (миграция 0041), плюс на каждой пер-тенантной таблице есть явная политика
   `tenant_isolation` по `tenant_id` (миграция 0042). Сервер ходит под
   `service-role` (обходит RLS by design), поэтому приложение работает как прежде;
   но любой НЕ-service доступ (публичный/анонимный ключ, будущая ограниченная
   роль) получает строки только своего тенанта, а по умолчанию - ноль строк.
   Чувствительные таблицы реестра (tenants/bots/invite_codes/pending_access)
   оставлены в режиме default-deny (без политики) - их не-service роли не читают.

Плюс кросс-тенант тесты и точечно закрытые места прошлых утечек (RPC
`match_expenses`, триггер аудита).

## Шифрование

- **API-ключи пользователей** (Anthropic, Groq) шифруются по схеме
  **envelope (KEK/DEK), AES-256-GCM** (`_shared/crypto_box.ts`, формат `v2:`):
  - **KEK (мастер-ключ)** хранится в **Supabase Vault** (секрет `finbot_kek_v1`),
    не в таблицах с данными и не в git. Edge достаёт его на серверном пути через
    RPC `get_kek` (только `service_role`).
  - **DEK (ключ данных)** - отдельный 256-битный ключ на каждого тенанта, хранится
    в БД в обёрнутом виде (зашифрован KEK) в таблице `tenant_deks`.
  - Поля шифруются DEK тенанта. Дамп БД показывает только шифртекст и обёрнутые
    DEK; без KEK из Vault их не расшифровать. Расшифровка - в памяти, в момент
    вызова ИИ.
  - **Крипто-шреддинг**: удаление DEK тенанта (`shredTenantKeys`) делает его данные
    нечитаемыми навсегда (основа для удаления по GDPR).
  - **Ротация**: смена KEK = переобёртывание только DEK; смена DEK = перешифрование
    данных только одного тенанта.
  - Legacy-формат `v1:` (один `KEY_ENC_SECRET` в Edge-секретах) ещё **читается**
    для совместимости, новые данные пишутся только в `v2:`.
- **Бэкапы** шифруются `age`.
- **Транспорт** - HTTPS везде (Telegram, Supabase, провайдеры ИИ).

## Доступ владельца (честная граница)

FinBot - это серверный бот: чтобы распознавать траты ИИ, показывать дашборд и
считать сводки, **сервер обязан читать финансовые данные в открытом виде**. Значит
тот, кто управляет сервером (владелец проекта Supabase), технически может получить
доступ к этим данным. Это неустранимое свойство любого серверного ИИ-сервиса.

Что **сделано**, чтобы максимально ограничить и обезопасить даже этот доступ:

- Секреты (API-ключи) зашифрованы и в открытом виде в БД не лежат - даже из дампа
  их не достать без `KEY_ENC_SECRET`.
- RLS закрывает прямой доступ к данным через публичный/анонимный ключ.
- Данные не попадают в git; бэкапы зашифрованы.
- Доступ к проду - только через серверные секреты.

Чего **нельзя** обещать без смены архитектуры: «владелец вообще не может видеть
финансовые данные». Для этого пришлось бы убрать серверный ИИ и дашборды и хранить
всё в зашифрованном виде только на устройстве пользователя - это другой продукт.
Настоящий zero-knowledge для серверного ИИ возможен только через confidential
computing / TEE (например, обработка в анклаве) - это потенциальное будущее, вне
текущего объёма.

## Ключи: бэкап и восстановление (P2.1)

- **KEK** живёт в Supabase Vault. Если потерять Vault/проект, все `v2`-поля
  (API-ключи) нечитаемы. Поэтому держи **офлайн-копию KEK** (sealed backup) в
  менеджере паролей. Достать текущий KEK: `select decrypted_secret from
  vault.decrypted_secrets where name='finbot_kek_v1'`.
- **`KEY_ENC_SECRET`** (legacy `v1:`) - в Edge-секретах и `.env` (gitignored).
- **age-ключ бэкапов** - в 1Password (показан один раз при сетапе).
- Восстановление и сценарий «БД жива, KEK утерян» описаны в
  `docs/INCIDENT_RESPONSE.md`.

## Что ещё стоит усилить (бэклог)

- Полное доменное включение RLS (P0.1 Барьер 2): отдельная НЕ-bypass роль БД +
  per-request tenant-claim. Заблокировано тем, что проект на новой системе ключей
  (асимметричный JWT, симметричного `SUPABASE_JWT_SECRET` нет) - нечем подписать
  `app_tenant`-токен. Варианты: включить legacy JWT-секрет, либо прямой Postgres
  (SET LOCAL) с переписыванием слоя данных. Пока политики `tenant_isolation` -
  подготовленный второй замок (service-role их обходит).
- Ротация KEK: переобёртывание DEK новым KEK (отдельный скрипт).
- GDPR: полное удаление всех данных тенанта по запросу (сейчас удаляются/шреддятся
  ключи; нужна полная очистка пер-тенантных таблиц) + политика retention.
- 2FA на админ-доступах (Supabase, GitHub, Telegram, менеджер секретов) - ручная
  настройка владельцем.
- HTTP-заголовки безопасности (HSTS и т.п.): GitHub Pages их не выставляет; нужен
  прокси/перенос хостинга. Сейчас на Mini App есть CSP через meta.

### Сделано

- CI-страж минимизирован: allowlist содержит только неустранимые файлы (резолв
  личности + кросс-тенант cron); конвертируемое переведено на `tenantDb` (setup-once).
- Включён + форсирован RLS на всех таблицах с данными (0041); добавлены явные
  политики `tenant_isolation` по `tenant_id` на пер-тенантные таблицы (0042).
- Envelope-шифрование (KEK в Vault + per-tenant DEK, формат `v2:`), крипто-шреддинг.
- Инструмент ротации ключей `scripts/rotate_keys.ts` (idempotent, resumable,
  `--dry-run`); миграция существующих ключей `v1:`->`v2:` выполнена.
- Append-only аудит-лог чувствительных операций `security_audit` (0044): key_set,
  key_deleted, crypto_shred, webhook_auth_fail, access_granted/revoked. UPDATE/DELETE
  отозваны у всех ролей приложения. Алерт админу при всплеске неудачных
  аутентификаций вебхука.
- Логи: скрабинг секретов/PII по имени поля и по значению, debug подавляется в проде.
- Управление ключами: `/apikey`/`/groqkey` (замена), `/delete_keys` (удаление +
  крипто-шреддинг DEK).
- CI: gitleaks (секреты, полная история) + Dependabot (github-actions).
- Rate-limit на чувствительные операции: экспорт CSV (`api-export`, 20/день) и
  операции с ключами (`/apikey`/`/groqkey`/`/delete_keys`, 15/день); экспорт
  пишется в аудит-лог.
